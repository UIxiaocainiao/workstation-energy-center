# 1. 文档用途

本文档用于规范《工位补能站》V1 原型图的标注方式，确保交互稿不只是“页面长什么样”，还能说明：

- 这个模块是干什么的
- 用户怎么操作
- 操作后会发生什么
- 边界情况怎么处理
- 前端和测试需要注意什么

适用于以下场景：

- 交互原型评审
- UI 视觉承接
- 前端开发实现
- 测试用例编写

# 2. 标注原则

## 2.1 一个模块必须说清 5 件事
每个模块至少说明：

1. **模块目的**
2. **展示内容**
3. **交互行为**
4. **状态变化**
5. **异常处理**

---

## 2.2 不只标“正常态”
原型标注必须补齐：

- 默认态
- hover / 点击态
- 选中态
- loading 态
- 成功态
- 失败态
- 空态
- 边界态

---

## 2.3 不写模糊话
避免这种写法：

- 点击后跳转
- 正常显示
- 有提示
- 出错提示用户

建议改成：

- 点击后滚动至“今日状态签到”模块，并在模块标题区域高亮 1.2 秒
- 成功后 toast 提示“今天也辛苦了”
- 失败后保留原输入内容，并展示“网络有点忙，点一次重试”

---

## 2.4 一条标注只说一件事
不要把多个规则混在一大段里。  
建议按“编号 + 短句”写，方便 UI、前端、测试逐条确认。

# 3. 原型标注结构模板

每个页面建议统一采用以下结构。

---

## 3.1 页面级标注模板

### 页面名称
例如：首页

### 页面目标
说明这个页面主要解决什么问题。  
例如：帮助用户快速理解产品定位并完成首次互动。

### 页面核心行为
列出用户在这个页面最重要的 1–3 个动作。  
例如：

1. 领取今日状态
2. 体验黑话翻译器
3. 浏览共鸣内容并抽一句补能

### 页面结构
列出从上到下的模块顺序。  
例如：

1. 顶部导航
2. 首屏 Banner
3. 今日状态签到
4. 下班倒计时
5. 首页简版黑话翻译器
6. 精选共鸣内容
7. 补能盲盒入口
8. 页脚

### 页面注意事项
说明该页面交互上最容易出问题的点。  
例如：

- 首屏不能信息过满
- CTA 必须足够突出
- 状态签到与翻译器要优先于内容浏览

---

## 3.2 模块级标注模板

### 模块名称
例如：今日状态签到

### 模块目的
说明为什么存在这个模块。  
例如：让用户低门槛完成一次情绪签到，建立复访习惯。

### 展示内容
列出模块包含哪些元素。  
例如：

- 模块标题
- 副文案
- 5 个状态按钮
- 签到反馈文案

### 交互规则
逐条写清用户操作逻辑。  
例如：

1. 用户点击任一状态按钮
2. 当前按钮高亮
3. 记录签到结果
4. 显示反馈文案
5. 支持当天覆盖更新

### 状态说明
例如：

- 默认态
- hover 态
- 选中态
- 更新成功态
- 提交失败态

### 异常处理
例如：

- 提交失败时，保留当前选中态
- toast 提示“网络有点忙，点一次重试”

### 前端说明
例如：

- 同一设备当天允许覆盖更新
- 默认使用本地缓存 + 接口记录
- 提交成功后不刷新页面

### 测试关注点
例如：

- 重复点击是否会异常提交
- 失败后是否保留选中态
- 成功后是否出现正确反馈

# 4.6 首页简版黑话翻译器标注模板

## 模块名称
首页简版黑话翻译器

## 模块目的
作为首页主传播钩子，让用户低门槛体验核心玩法。

## 展示内容
- 模块标题
- 模式切换
- 输入框
- 快捷示例
- 生成按钮
- 结果卡片
- 复制 / 再来一句 / 打开完整翻译器

## 模式
- 老板黑话 → 真实含义
- 真实吐槽 → 体面表达

## 交互规则
### TR-01
默认选中“老板黑话 → 真实含义”。

### TR-02
点击示例标签后，自动填充输入框。

### TR-03
未输入时，按钮文案显示“先选一句示例”。

### TR-04
输入有效内容后，按钮文案切换为“立即翻译”。

### TR-05
点击后进入 loading 态，按钮禁用。

### TR-06
返回结果后展示结果卡片，并出现两个主动作：
- 复制结果
- 再来一句

### TR-07
“打开完整翻译器”作为辅助动作，不抢主按钮层级。

### TR-08
切换模式时不清空输入，只更新 placeholder 和示例。

## 输入校验
### TR-09
小于 2 字提示：“再多说一点，我才能更懂你”。

### TR-10
超过 100 字提示：“最多输入 100 字，先抓重点”。

## 状态说明
- 默认态
- 输入态
- 空输入态
- loading 态
- 结果成功态
- 无匹配兜底态
- 复制成功态
- 接口失败态

## 异常处理
### TR-11
接口失败时保留输入内容和当前模式。

### TR-12
提示文案：“网络有点忙，点一次重试”。

## 前端说明
- 模式切换不清空输入
- 结果区高度建议固定，避免页面跳动
- 再来一句优先返回未出现过结果

## 测试关注点
- 示例填充是否正确
- 模式切换是否保留输入
- 输入边界值是否正确提示
- loading / 成功 / 失败状态是否完整

# 6. 状态标注规范

建议在原型图中统一用以下方式标注状态：

- **[D]** 默认态
- **[H]** Hover 态
- **[S]** 选中态
- **[L]** Loading 态
- **[OK]** 成功态
- **[ERR]** 错误态
- **[EMPTY]** 空态
- **[EDGE]** 边界态

示例：

- `TR-05 [L] 点击生成后按钮进入 loading 态，文案变为“翻译中...”`
- `S-07 [ERR] 提交失败时保留当前选中态，并提示“网络有点忙，点一次重试”`

# 10. 原型评审使用方法

在原型评审时，建议按以下顺序逐页走查：

1. 页面目标是否明确
2. 首屏是否清晰
3. 主路径是否顺
4. 状态是否补齐
5. 异常是否可恢复
6. 文案是否统一
7. 是否可交给 UI
8. 是否可交给研发
9. 是否可交给测试

---

# 11. 结论

交互原型标注稿不是“把线框图说明一下”，而是把用户每一步怎么走、看到什么、做什么、失败了怎么办、成功了下一步是什么，都写清楚。

对《工位补能站》V1 来说，标注稿最重要的目标是：

- 让 UI 不猜
- 让前端不问
- 让测试不漏
- 让体验不丢
